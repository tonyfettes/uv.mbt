# Build System of `uv.mbt`

The build process of `libuv` inside `uv.mbt` is design with following goal:

1. Zero-configuration. There nothing to configure beyond adding the module as a
   dependency.
2. TinyCC compatible. Using `tcc` ensures a blazing fast test feedback for
   prototyping.
3. Safety. No script/command will be run upon/after installation.

To achieve these goals, `uv.mbt` pre-processes all C files of `libuv`, so that
they can be compiled and linked with no extra command line arguments to
compiler. However, this only works on macOS platform, as on both Linux and
Windows `libuv` needs to be linked against some libraries to work. As we are
working on supporting these two platforms, we might have to compromise some of
our goals above.

## Rationale

There are plenty of ways to build a C library using `moon`.

1. **Pre-build tasks**. It is a generic hook that allows user to run arbitrary
   command before the building step of `moon`. However, to have the C file
   generated by MoonBit to be linked against the library built using pre-build
   tasks, the `"moon.pkg.json"` has to be changed to contains link flags that
   tells `moon` to link against their artifacts to the built library. Also,
   these link flags fails to pass down along the depedency chain, which means the
   end users will have to manually specify the link flags. Manual specification
   is not only tedious, but also error-prone, and harder for upstream library to
   push a update on build process to users.
2. **Out-sourcing to other build system**. This method integrates with existing
   build system well, as you can set the `"cc"` to be `"echo"` to prevent
   `moon` from actually building C files generated by the compiler. However, it
   is likely that in the future MoonBit toolchain `moon` will not emitting C
   files, making this method no longer a good choice for long-term project
   development.
3. **(New) Post-add tasks**. This is a new hooks in `moon` that can run
   arbitrary command upon dependency installation. This mechanism is promising
   and will potentially solve `uv.mbt`'s linking on Linux and Windows, and
   potentially can serve as an entry point for binary-distribution and integration
   with existing build systems.
4. **Pre-processing**. This is the approach we are using now. By handling all
   dirty work ahead, this method provides maximum user experience. However,
   this method requires the C dependency to be 100% C, as only C files can be
   built by `moon` without specify any flags.

## Implementation

Basically, the pre-processing of C files does the following 4 things:

1. Flatten the source hierarchy.
2. Prepend macro definitions to every file.
3. Relocate `#include`.
4. Update `moon.pkg.json`.

### Flatten Hierarchy

In MoonBit, the path of package _is_ its relative path to the module.
Therefore, if we were keeping the file hierarchy of `libuv`, we have to create
multiple packages and reference them in the root package. In most of the case,
these packages are just empty, other than the C files built as stubs.
Therefore, the MoonBit compiler will complain that these packages are not used.

To eliminate these warnings, we flatten the file hierarchy of `libuv`, so that
every source, every header are place at the same level inside the root package.
However, there might be two files from different folders that has the same
`basename`. For example, `src/win/internal.h` and `src/unix/internal.h`. To solve
the name clashing, we encoding the path of the file, by replacing the path
separator `/` to `#` as the new `basename` for the file.

- `src/win/internal.h` gets renamed to `src#win#internal.h`
- `src/unix/internal.h` gets renamed to `src#unix#internal.h`
